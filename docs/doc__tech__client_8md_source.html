8 Program work flow (#client_workflow)
9 ====================================
10 - Parse terminal arguments
11 - Get id server address to check if it exists:
12 - Initiates the program fundamental variables (See init_program())
13  + Initialize sockets (\ref file_descriptors_client)
14  + Initializes the timer implementation
15  + Fetch servers from the identity server
16  + Saves the servers in the list of message servers
17  + Selects one of the servers to communicate with
18 - Enters a interactive loop where the program will run until it finds an error or is told to exit.
19  + Sends all the file descriptors (fd) to a set, the set will be watched and each fd will be handled only when its ready.
20  + If no server is selected, a specific error occurred or a selected server isn't answering we must get a new and valid one, only then the program proceeds. To better understand how we handled the matters with bad servers and functional ones see [Server Validity Check](\ref server_validity_check).
21  + The program will wait in the select function until one of the [file descriptors](\ref file_descriptors_client) is ready to be read.
22  + If the timer ticks the [test for server validity](\ref server_validity_check) is handled.
23  + If an incoming udp message is received the incoming message is handled. See [udp handling](\ref udp_handle_client).
24  + If the user sends input to console the command must be interpreted. See [user input handling](\ref user_input_client).
25  + Strings are cleaned (filled with '\0'), and prompt is reprinted.
26  + Loop restarts.
27 - Free all allocated memory and exit.

34  + Outgoing fd: This socket is only used to ask for servers from the identity server

39 Server Validity Check {#server_validity_check}
40 ============================================
41 ###The if block
42 In order to explain better the work that is done in this block, there is a need to know how the server ban implementation works.
43 It's based in two major calls:
44 > ban_server(list, server) - Puts the server in a list of servers who are banned, and sets it's ban time to a specified time. If it already exists restore the ban time to the specified time.\n
45 > is_banned(list, server) - Checks if server is banned, returns true if it's banned and false if not. The ban time counter on that server is decreased.\n
46 
47 If a server is not answering the server is banned and removed from the messages servers list. Then starts the look for a new server that is not banned, if we find one that is already banned it is removed from the message servers list. Until we find a non banned server or there aren't no servers left in the list.
48 
49 On the end of the block one of two cases will happen:
50 - 1: We find a server and exit the block with that server.
51 - 2: The list is empty and new servers list is fetched from the identity server and the block restarts to check the fetched servers.
52 
53 If the servers fetched are still the same after every fetch the banned servers will became unbanned eventually and can be used after that.

55 ###The test
56 The program only uses the block after errors on sending outgoing communications to servers.\n
57 If there isn't a server selected to work.\n
58 Or if the server didn't passed the test in the specified time.
59 
60 Now how does the program states that the server is not working.
61 After each user request to the server a test is performed. Based on the ask_for_messages() call, we can determine if the server is answering or not.
62 
63 So after a publish() is called to perform a user request a ask_for_messages() is also called with a parameter who sets the no need to print the answer when it comes. In the mean time the test is scheduled with a call to ask_server_test() and a internal flag is set.\n
64 The timer is responsible to check whether the server answered or not.
65 
66 Every time the timer is triggered the function exec_server_test() is called and if ask_server_test() was called the "counter" inside exec_server_test() is incremented, if it happened two times the server is considered not_answering.
67 
68 If a server answers with a valid response, the flag is reset and the next timer trigger will clean the test and the server is not marked as unresponsive.
69 
70 To test after a user call to ask_for_messages() the same is done, only the parameter who states if the result is to print or not is set to print output.

72 Udp incoming messages treatment {#udp_handle_client}
73 =====================================================
74 The client is projected to only be able to receive one type of message from the messages servers. the messages carries one header: `'MESSAGES\n'` and can contain multiple lines after that containing info on each message.
75 
76 First thing to do is allocate some space in memory to save our incoming communication, being sent via UDP, the message arrives all at once and the buffer needs to have size for the entire communication.\n
77 The size is calculated knowing the number of messages that were requested to the server. If the server sends more, some info may be lost.
78 
79 After receiving the information, it is printed to the user. With a little aesthetic modification.

81 User input interpretation {#user_input_client}
82 ===============================================
83 The commands that the user can input are:
84 These are not case-sensitive and there are numeric shortcuts.
85 Command |Shortcut
86 :-----------------------------|:-------:
87 show\_servers | 0
88 publish __message__ | 1
89 show\_latest\_messages __n__ | 2
90 show\_selected\_server | 3
91 exit | 9
92 
93 To interpret the user request the function `scanf("%s%*[ ]%140[^\t\n]", a, b)` is called.\n
94 The first word until it finds a white-space is saved in 'a'.\n
95 Then we throw away the white-space.\n
96 And finally the next 140 characters, if not found tab or new-line, are saved in 'b'.\n
97 For comparing the commands with the user_input strcasecmp() is used to compare without case sensitive characters.
98 
99 If the user inputs a message with more than 140 characters, the remaining is discarded.\n
100 The user is asked if he really wants to send the sliced message, if the operation in 'a' is publish.\n
101 See publish() to know more on how are messages sent.\n
102 See ask_for_messages() to see how the requests are made.
103 
104 The show_servers command prints the list currently being used to select the server at work.\n
105 Exit command breaks out of the loop.
