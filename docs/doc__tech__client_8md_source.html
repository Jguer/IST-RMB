2 ------------------------------------

7 Program work flow (#client_workflow)
8 ====================================
9 - Parse terminal arguments
10 - Get id server address to check if it exists:
11 - Initiates the program fundamental variables (See init_program())
12  + Initialize sockets (\ref file_descriptors_client)
13  + Initializes the timer implementation
14  + Fetch servers from the identity server
15  + Saves the servers in the list of message servers
16  + Selects one of the servers to communicate with
17 - Enters a interactive loop where the program will run until it finds an error or is told to exit.
18  + Sends all the file descriptors (fd) to a set, the set will be watched and each fd will be handled only when its ready.
19  + If no server is selected, a specific error occurred or a selected server isn't answering we must get a new and valid one, only then the program proceeds. To better understand how we handled the matters with bad servers and functional ones see [Server Validity Check](\ref server_validity_check).
20  + The program will wait in the select function until one of the [file descriptors](\ref file_descriptors_client) is ready to be read.
21  + If the timer ticks the [test for server validity](\ref server_validity_check) is handled.
22  + If an incoming udp message is received the incoming message is handled. See [udp handling](\ref udp_handle_client).
23  + If the user sends input to console the command must be interpreted. See [user input handling](\ref user_input_client).
24  + Strings are cleaned (filled with '\0'), and prompt is reprinted.
25  + Loop restarts.
26 - Free all allocated memory and exit.

28 File descriptors{#file_descriptors_client}
29 =========================================
30 - STDIN: This file descriptor is used to read from the user and is only ready when the user send input to the terminal.
31 - UDP:
32  + Binded fd: This socket is binded to a port in the system, so that the servers can communicate to a well known port and the select function can process the file descriptor
33  + Outgoing fd: This socket is only used to ask for servers from the identity server
34 - TIMER: (Library: timer_fd.h) This timer is implemented via a file descriptor. When the timer is triggered the file descriptor is set as ready to read.

36 The file descriptors are handled with the select() function.

38 Server Validity Check {#server_validity_check}
39 ============================================
40 ###The if block
41 In order to explain better the work that is done in this block, there is a need to know how the server ban implementation works.
42 It's based in two major calls:
43 > ban_server(list, server) - Puts the server in a list of servers who are banned, and sets it's ban time to a specified time. If it already exists restore the ban time to the specified time.\n
44 > is_banned(list, server) - Checks if server is banned, returns true if it's banned and false if not. The ban time counter on that server is decreased.\n
45 
46 If a server is not answering the server is banned and removed from the messages servers list. Then starts the look for a new server that is not banned, if we find one that is already banned it is removed from the message servers list. Until we find a non banned server or there aren't no servers left in the list.
47 
48 On the end of the block one of two cases will happen:
49 - 1: We find a server and exit the block with that server.
50 - 2: The list is empty and new servers list is fetched from the identity server and the block restarts to check the fetched servers.
51 
52 If the servers fetched are still the same after every fetch the banned servers will became unbanned eventually and can be used after that.

54 ###The test
55 The program only uses the block after errors on sending outgoing communications to servers.\n
56 If there isn't a server selected to work.\n
57 Or if the server didn't passed the test in the specified time.
58 
59 Now how does the program states that the server is not working.
60 After each user request to the server a test is performed. Based on the ask_for_messages() call, we can determine if the server is answering or not.
61 
62 So after a publish() is called to perform a user request a ask_for_messages() is also called with a parameter who sets the no need to print the answer when it comes. In the mean time the test is scheduled with a call to ask_server_test() and a internal flag is set.\n
63 The timer is responsible to check whether the server answered or not.
64 
65 Every time the timer is triggered the function exec_server_test() is called and if ask_server_test() was called the "counter" inside exec_server_test() is incremented, if it happened two times the server is considered not_answering.
66 
67 If a server answers with a valid response, the flag is reset and the next timer trigger will clean the test and the server is not marked as unresponsive.
68 
69 To test after a user call to ask_for_messages() the same is done, only the parameter who states if the result is to print or not is set to print output.

71 Udp incoming messages treatment {#udp_handle_client}
72 =====================================================
73 The client is projected to only be able to receive one type of message from the messages servers. the messages carries one header: `'MESSAGES\n'` and can contain multiple lines after that containing info on each message.
74 
75 First thing to do is allocate some space in memory to save our incoming communication, being sent via UDP, the message arrives all at once and the buffer needs to have size for the entire communication.\n
76 The size is calculated knowing the number of messages that were requested to the server. If the server sends more, some info may be lost.
77 
78 After receiving the information, it is printed to the user. With a little aesthetic modification.

80 User input interpretation {#user_input_client}
81 ===============================================
82 The commands that the user can input are:
83 These are not case-sensitive and there are numeric shortcuts.
84 Command |Shortcut
85 :-----------------------------|:-------:
86 show\_servers | 1
87 publish __message__ | 2
88 show\_latest\_messages __n__ | 3
89 show\_selected\_server | 4
90 exit | 9
91 
92 To interpret the user request the function \code{C} scanf("%s%*[ ]%140[^\n]", a, b) \endcode is called.\n
93 The first word until it finds a white-space is saved in 'a'.\n
94 Then we throw away the white-space.\n
95 And finally the next 140 characters, if not found new-line, are saved in 'b'.\n
96 For comparing the commands with the user_input strcasecmp() is used to compare without case sensitive characters.
97 
98 There's protection to escape characters input, who prevent that messages are sent with possibly dangerous to program characters.
99 
100 If the user inputs a message with more than 140 characters, the remaining is discarded.\n
101 The user is asked if he really wants to send the sliced message, if the operation in 'a' is publish.\n
102 See publish() to know more on how are messages sent.\n
103 See ask_for_messages() to see how the requests are made.
104 
105 The show_servers command prints the list currently being used to select the server at work.\n
106 Exit command breaks out of the loop.
